# 文档管理 Skill

执行项目文档标准化管理流程，确保文档分类正确、操作合规。

---

## 标准文档 Frontmatter

所有文档必须在开头包含 YAML frontmatter：

```yaml
---
title: 文档标题                    # 简短明确的功能/模块名称
type: prd                          # 文档类型，常见：prd | arc | dev | next | todo | architecture，也支持其他自定义类型
status: draft                      # draft | active | archived
created: 2026-03-23
updated: 2026-03-23
description: 简短描述核心重点
---
```

### Type 说明

type 用于标识文档性质，常见类型：

| Type | 用途 | 存放路径 |
|------|------|---------|
| `prd` | 产品需求文档 | `Doc/{功能名}_prd.md` |
| `arc` | 架构设计文档 | `Doc/{功能名}_arc.md` 或 `Doc/Architecture/*.md` |
| `dev` | 开发实现文档 | `Doc/{功能名}_dev.md` |
| `next` | 下一步计划 | `Doc/{功能名}_next.md` |
| `todo` | 全局待办 | `Doc/_todo.md` |
| `architecture` | 基础架构文档 | `Doc/Architecture/*.md` |

**也支持其他自定义类型**，根据实际需求定义。

### Status 说明

| Status | 说明 |
|--------|------|
| `draft` | 草稿，编写中 |
| `active` | 已确定，开发依据 |
| `archived` | 已归档，功能下线或废弃 |

**注意**：文档不再需要时直接删除，去 git 记录中查看历史。不要保留无用的文档。

## 操作流程（强制顺序执行）

### 步骤 1：判定文档类别

根据用户意图，确定文档类型和路径：

| 判定条件 | Type | 路径 |
|---------|------|------|
| 影响全局基础设施（数据库、网络、存储等） | architecture | `Doc/Architecture/` |
| 描述"要做什么"、需求背景 | prd | `Doc/` |
| 描述技术方案、模块设计 | arc | `Doc/` |
| 记录实现问题和解决方案 | dev | `Doc/` |
| 记录未完成工作 | next | `Doc/` |
| 全局待办 | todo | `Doc/` |

**基础架构先行原则**：功能设计前，必须先查阅相关基础架构文档。

---

### 步骤 2：扫描文档库

**目的**：发现相关文档、检查冲突、建立上下文。

**执行**：
1. 根据确定的路径，扫描目标文件夹及其子文件夹
2. 读取所有文档的 frontmatter（不读正文）
3. 通过 `title`、`type`、`description` 筛选相关文档

**输出**：相关文档清单（含 type、status、description）

---

### 步骤 3A：新建文档流程

**必须回答**："为什么该文档不存在？"

有效理由：
- 该功能首次进入此阶段
- 前期未文档化，现补建
- 功能拆分后独立出来

**执行**：
1. 创建文件，添加标准 frontmatter
2. 根据 type 选择模板填充内容
3. 在正文中建立 Obsidian 双向链接 `[[文件名]]`

---

### 步骤 3B：修改文档流程

**必须回答**："为什么需要修改？当前存在什么问题？"

有效理由：
- 需求变更：新增验收标准
- 架构调整：接口方案变化
- 开发踩坑：记录问题解决方案

**执行**：
1. 读取现有文档 frontmatter
2. 更新 `updated` 时间
3. 如 status 变化，同步更新
4. 执行内容修改
5. **评估影响范围**：如涉及基础架构变更，列出所有受影响功能

---

### 步骤 3C：删除文档流程

**原则**：不再需要的文档直接删除，去 git 记录中查看历史。

**执行**：
1. 确认文档无用或已被替代
2. 检查是否有其他文档引用此文档
3. 如有引用，更新相关文档中的链接
4. 删除文档

---

## 关键检查点

任何文档操作后，执行自检：

- [ ] frontmatter 是否完整（5 个必需字段）？
- [ ] `type` 是否正确？
- [ ] `status` 是否符合当前阶段？
- [ ] 正文是否有对应的 `[[文件名]]` 双向链接？
- [ ] 如涉及修改，是否评估了影响范围？
