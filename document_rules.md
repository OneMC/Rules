# 文档规则

> 版本：v1.4  
> 原则：按阶段建档，基础架构先行，双向关联追踪，只留最新版本

---

## 一、文档分类与位置

### 1.1 项目基础架构文档

位于 `Doc/Architecture/`，定义项目全局基础架构，**通常没有对应 PRD**，所有功能必须遵守。

```
Doc/Architecture/
├── Database_arc.md         # 数据库设计（表结构、索引、迁移策略）
├── Network_arc.md          # 网络层设计（请求封装、拦截器、错误处理）
├── Storage_arc.md          # 存储架构（缓存策略、文件管理）
└── ...                     # 其他全局基础设施
```

**特点：**
- 无对应 PRD，属于技术决策
- 修改时必须评估全局影响

### 1.2 功能文档

位于 `Doc/`，按功能模块组织：

```
Doc/
├── {功能名}_prd.md       # 产品需求文档
├── {功能名}_arc.md       # 架构设计文档
├── {功能名}_dev.md       # 开发实现文档
└── {功能名}_next.md      # 下一步计划（功能未完全交付时使用）

示例：
├── UserProfile_prd.md
├── UserProfile_arc.md
├── UserProfile_dev.md
└── UserProfile_next.md
```

### 1.3 开发辅助文档

用于管理开发过程中的临时状态和连续性，位于 `Doc/`，与功能文档并列：

| 文档类型 | 文件名 | 用途 | 生命周期 |
|----------|--------|------|----------|
| **TODO 列表** | `_todo.md` | 开发中的临时待办（跨功能、跨阶段） | 条目完成即删 |
| **下一步计划** | `{功能名}_next.md` | 记录功能未完成部分，保证开发连续性 | 功能完成后删除 |

```
Doc/
├── _todo.md                # 开发待办列表（临时）
├── UserProfile_next.md     # 用户资料功能的下一步计划
└── ...
```

## 二、核心原则：按阶段建档，只留最新

- **需要什么建什么** —— 需求阶段写 PRD，架构阶段写 ARCH，开发阶段写 DEV，可裁剪组合
- **只保留最新版本** —— 旧逻辑直接覆盖，历史在 Git 里查
- **文档名 = 功能名_类型** —— 一目了然

---

## 三、三种文档职责

| 文档 | 用途 | 谁写 |
|------|------|------|
| `_prd.md` | 需求背景、功能描述、验收标准 | 产品/开发 |
| `_arc.md` | 技术方案、模块设计、接口定义 | 开发 |
| `_dev.md` | **开发中遇到的问题及解决方案** | 开发 |

## 四、文档操作流程（强制）

**任何文档操作前，必须依次执行以下判断：**

### 步骤 1：判定文档类别

首先判定当前工作属于哪类文档：

| 文档类别 | 判定标准 | 存放路径 | 文档要求 |
|----------|----------|----------|----------|
| **基础架构文档** | 影响全局基础设施（数据库、网络、安全、存储等） | `Doc/Architecture/` | 不需要 PRD，直接从 ARCH 开始 |
| **功能文档** | 针对具体功能模块 | `Doc/` | 按阶段写 PRD/ARCH/DEV |

**基础架构先行原则：** 功能设计前，必须先查阅相关基础架构文档，确保方案符合架构约束。

### 步骤 2：确定工作类型

根据当前工作内容，选择文档类型：

| 工作类型 | 判定标准 | 文档后缀 |
|----------|----------|----------|
| PRD | 描述"要做什么"、需求背景、验收标准 | `_prd.md` |
| ARCH | 描述"技术方案、模块设计、接口定义" | `_arc.md` |
| DEV | 描述"代码实现、踩坑记录、问题解决方案" | `_dev.md` |
| OTHER | 不属于以上三类（如纯工具函数使用说明） | 按情况取名 |

### 步骤 3：扫描文档库

根据文档类别确定搜索路径：

```
IF 文档类别 == "基础架构文档":
    搜索路径 = Doc/Architecture/
ELSE:
    搜索路径 = Doc/
```

在对应路径下搜索目标文件：
- **功能文档** (`{功能名}_{类型}.md`) 和 **下一步计划** (`{功能名}_next.md`)：按功能名搜索
- **TODO 列表** (`_todo.md`)：固定文件名，直接检查是否存在

搜索结果：
- **存在** → 进入【修改流程】
- **不存在** → 进入【新建流程】

### 步骤 4A：新建文档

必须回答：**"为什么该文档不存在？"**

有效理由示例：
- "该功能首次进入此阶段"
- "前期未文档化，现补建"
- "功能拆分后独立出来"

**基础架构文档新建额外要求：**
- 必须说明该架构解决什么全局问题
- 必须列出受约束的功能范围
- 必须在相关功能文档中添加反向链接（见第五章）

然后：
1. 选择对应模板（见第五章）
2. 创建文件并初始化
3. 写入内容到对应章节
4. **建立 Obsidian 双向链接（见第五章）**

### 步骤 4B：修改文档

必须回答：**"为什么需要修改？当前存在什么问题？"**

有效理由示例：
- "需求变更：新增第 3 个验收标准"
- "架构调整：接口从 REST 改为 GraphQL"
- "开发踩坑：记录问题 #5 的解决方案"

**基础架构文档修改额外要求：**
- 必须评估影响范围，列出所有受影响的功能
- 必须同步修改所有受影响功能的对应文档
- 必须在修改理由中注明：「因 Architecture/XXX_arc.md 变更，同步调整」

然后：
1. 精确定位章节（必须明确指出修改哪一级标题）
2. 执行内容修改
3. 更新文档元信息（如最后修改时间）
4. **更新 Obsidian 双向链接，检查关联完整性**

### 例外判定：简单功能是否需要文档？

| 条件 | 判定 |
|------|------|
| 代码行数 > 50 行 | 需要 DEV 文档 |
| 涉及 2 个以上模块交互 | 需要 ARCH 文档 |
| 有任何验收标准 | 需要 PRD 文档 |
| 纯工具函数，无复杂逻辑 | 可省略，但需代码注释说明 |

## 五、Obsidian 双向关联规范

使用 Obsidian 的 `[[文件名]]` 语法建立文档间双向链接，确保影响范围可追溯。当发现新功能与已有功能存在逻辑冲突时，必须同步更新相关文档，并创建待办事项跟踪修复。

### 5.1 链接规则

| 场景 | 链接方向 | 示例 |
|------|----------|------|
| 功能依赖基础架构 | 功能文档 → 架构文档 | `UserProfile_arc.md` 中写 `[[Database_arc]]` |
| 功能间调用关系 | 调用方 → 被调用方 | `Order_arc.md` 中写 `[[Payment_arc]]` |
| 架构修改影响功能 | 架构文档 → 受影响功能 | `Database_arc.md` 中写 `[[UserProfile_arc]]` |

### 5.2 必须建立链接的时机

1. **新建基础架构文档时** —— 在文档底部「相关功能」章节列出所有受影响功能
2. **新建功能文档时** —— 在「依赖架构」章节引用相关基础架构文档
3. **修改基础架构时** —— 检查并更新所有受影响功能的链接关系

## 六、文档参考模板

> 该部分模板仅供参考，各类型文档需在写时按照详细规则进行

### 6.1 PRD 模板

```markdown
# 功能：用户资料

## 需求背景
用户需要查看和修改个人资料。

## 功能描述
- 展示头像、昵称、手机号
- 支持修改头像和昵称

## 验收标准
- [ ] 头像上传成功并显示
- [ ] 昵称修改后保存生效
```

### 6.2 ARCH 模板

```markdown
# 架构：用户资料

## 技术方案
采用 MVVM + Combine

## 模块设计
ProfileViewController → ProfileViewModel → UserService

## 接口定义
- `GET /api/v1/user/profile`
- `POST /api/v1/user/profile`

## 数据模型
struct UserProfile {
    let avatarURL: String
    let nickname: String
}
```

### 6.3 DEV 模板（重点：问题与解决）

```markdown
# 开发：用户资料

## 实现要点
- 头像使用 Kingfisher 加载
- 昵称限制 20 个字符

## 问题与解决

### 问题 1：头像上传后缓存未刷新
**现象**：上传新头像后，UI 仍显示旧头像  
**原因**：Kingfisher 缓存了旧 URL  
**解决**：上传成功后清除缓存并重新加载
```swift
KingfisherManager.shared.cache.removeImage(forKey: avatarURL)
imageView.kf.setImage(with: newURL)
```

## 七、开发过程管理

### 7.1 TODO 管理

TODO 属于**临时待办**，记录在 **`Doc/_todo.md`**，可跨功能、跨阶段。

**适用场景：**
- 跨多个功能的待办（如"统一所有页面的错误处理"）
- 新功能的初步想法（尚未创建功能文档）
- 技术债和重构任务
- 临时中断的任务记录

**规则：**

1. **格式**
   ```markdown
   # TODO

   ## 高优先级
   - [ ] 修复 Database_arc 中的索引设计问题
   - [ ] UserProfile 头像上传接口联调

   ## 中优先级
   - [ ] 调研新的图片压缩库
   - [ ] Order 和 Payment 的异常处理统一

   ## 低优先级 / 想法
   - [ ] 考虑引入 SwiftLint 规范代码风格
   ```

2. **使用规范**
   - 开发中随时更新 TODO 列表
   - 任务完成后**立即删除**对应条目（不要打勾保留）
   - 不需要保留历史，不需要写完成时间
   - 与功能强相关的 TODO 转移到该功能的 `_next.md` 文档

3. **禁止事项**
   - 禁止将 TODO 作为长期备忘录
   - 禁止已完成的 TODO 打勾保留
   - 禁止在 TODO 中记录已创建功能文档的需求（应转为 PRD）

### 7.2 下一步计划文档（`{功能名}_next.md`）

当功能**本次未完全交付**时必须创建，记录剩余工作保证开发连续性。

**模板：**

```markdown
# 下一步：用户资料

## 当前进度
头像上传、昵称修改已完成，手机号绑定待开发

## 剩余任务
1. 手机号绑定流程
   - 手机号格式校验
   - 短信验证码接入
   - 换绑逻辑

2. 资料隐私设置
   - 公开/隐藏开关
   - 黑名单处理

## 阻塞项
- 等待短信服务商接口文档

## 预计继续时间
2024-03-25
```

**规则：**

1. **创建时机**
   - 开发任务未完全完成时
   - 需要中断当前工作转做其他任务时
   - 复杂功能分多阶段交付时

2. **更新时机**
   - 每次开发结束必须更新进度
   - 下次开发前必须先阅读此文档
   - 进度有变化时立即更新

3. **结束时机**
   - 功能全部完成后**立即删除**该文档
   - 或移动到 `Doc/Archive/` 归档（可选）

4. **适用范围**
   - 功能文档（PRD/ARCH/DEV）均可有对应的 next 文档
   - 基础架构文档分阶段实施时同样适用

## 八、必须遵守的原则-总结
1. **操作前强制判定** —— 写文档前必须先确定工作类型、查库、说明新建或修改理由
2. **基础架构先行** —— 功能设计前必须先查阅并遵守相关基础架构文档
3. **按阶段建档** —— 到什么阶段写什么文档，可裁剪但不可跳流程
4. **只留最新** —— 旧逻辑直接删改，历史版本在 Git 里
5. **影响范围检查** —— 修改功能前，先评估是否影响其他已有功能；如有影响，必须同步更新受影响功能的对应文档，并在修改理由中说明关联关系
6. **架构修改联动** —— 修改基础架构时，必须同步更新所有受影响功能文档，并在修改理由中注明关联关系
7. **双向关联追踪** —— 所有文档必须建立 Obsidian 双向链接，确保影响范围可追溯
8. **代码和文档同 PR** —— 改功能必须同时改对应文档
9. **DEV 必写问题与解决** —— 踩过的坑要记下来，方便后人（和自己）
