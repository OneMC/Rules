# Document Prompt

## 作用
指导 AI 在开发过程中正确管理项目文档，包括 PRD、架构文档、DEV 文档的创建和维护。

## 配置方式
在 AI 工具中配置，用于涉及文档创建或修改的场景。

## 何时激活
- 创建新功能时
- 修改现有功能时
- 进行架构设计时
- 记录开发问题和解决方案时

## 核心指令

### 1. 文档判定流程

**步骤 1：判定文档类别**

| 文档类别 | 判定标准 | 存放路径 |
|----------|----------|----------|
| 基础架构 | 影响全局（数据库、网络、安全、存储） | `Doc/Architecture/` |
| 功能文档 | 针对具体功能模块 | `Doc/` |

**步骤 2：确定工作类型**

| 工作类型 | 文档后缀 | 内容要求 |
|----------|----------|----------|
| PRD | `_prd.md` | 需求背景、功能描述、验收标准 |
| ARCH | `_arc.md` | 技术方案、模块设计、接口定义 |
| DEV | `_dev.md` | 问题与解决方案 |

### 2. 强制判定问题

**新建文档时必须回答：**
> "为什么该文档不存在？"
- 有效理由：功能首次进入此阶段、前期未文档化现补建、功能拆分后独立

**修改文档时必须回答：**
> "为什么需要修改？当前存在什么问题？"
- 有效理由：需求变更、架构调整、开发踩坑记录

### 3. 文件命名规范

```
Doc/
├── Architecture/
│   ├── Database_arc.md
│   └── Network_arc.md
├── UserProfile_prd.md
├── UserProfile_arc.md
├── UserProfile_dev.md
└── UserProfile_next.md  # 功能未完成时使用
```

### 4. Obsidian 双向链接

**必须建立链接的场景：**
- 新建基础架构文档 → 列出受影响功能
- 新建功能文档 → 引用依赖的架构文档
- 修改基础架构 → 更新所有受影响功能

**链接语法：**
```markdown
## 依赖架构
- [[Database_arc]]
- [[Network_arc]]

## 相关功能
- [[UserProfile_arc]]
```

### 5. 检查清单

创建/修改文档后验证：
- [ ] 文档类别判定正确（架构 vs 功能）
- [ ] 文件命名符合 `{功能}_{类型}.md` 格式
- [ ] 已说明新建/修改理由
- [ ] 已建立 Obsidian 双向链接
- [ ] 代码和文档在同一 PR 中提交
