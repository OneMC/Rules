# PRD Skill - 产品需求专业编写

提供标准化、渐进式 PRD 编写流程。严格遵循 `document_rules.md` 文档规范。

---

## 执行流程概览

PRD 编写分 **7 个阶段** 顺序执行，每个阶段产出独立文档，通过更新推进，禁止一次完成全部：

| 阶段 | 文档 | 目的 | 是否暂停 |
|-----|------|------|---------|
| 0 | 00-context.md | 扫描项目上下文 | 否 |
| 1 | 01-requirement.md | 分析需求，判定新建/修改 | 是（需澄清） |
| 2 | 02-research.md | 调研并产出可选方案 | 是（需选择） |
| 3 | 03-draft-prd.md | 编写 PRD 草稿 | 否 |
| 4 | 04-review.md | 自我 Review | 是（重大问题） |
| 5 | 05-impact.md | 影响评估与关联修改 | 是（breaking change） |
| 6 | 06-final-review.md | 统一 Review | 否 |
| 7 | 07-summary.md | 正式输出与总结 | 否 |

---

## 核心原则

1. **渐进式完成** —— 每个阶段独立成文档，通过更新推进，禁止一次完成全部
2. **项目上下文优先** —— 必须分析现有 PRD/ARCH，理解项目现状后再设计
3. **方案对比** —— 视需求复杂度，搜索相关知识，产出合适的可选方案
4. **影响评估必做** —— 任何方案必须评估对其他功能的影响
5. **双向链接追踪** —— 使用 Obsidian `[[文件名]]` 语法建立文档关联

---

## 阶段详解

### 阶段 0：项目扫描（前置步骤）

**目的**：建立项目上下文，避免设计冲突；发现关联任务和未完成工作。

**执行**：
1. 扫描 `Doc/Architecture/` 及其子文件夹，通过文档标题和简介筛选与当前需求相关的架构文档
2. 扫描 `Doc/` 及其子文件夹，通过文档标题和简介筛选相关的 `*_prd.md` 和 `*_arc.md`
3. 检查 `Doc/_todo.md`（全局待办）和 `{功能名}_next.md`（该功能的下一步计划）：
   - **目的**：发现是否已有相关待办事项或未完成的关联功能，避免重复规划或冲突
   - 如有相关内容，纳入上下文考量
4. 输出 `00-context.md`：
   - 项目当前阶段
   - 已有相关功能简要描述
   - 相关基础架构约束
   - 潜在冲突点

---

### 阶段 1：需求分析

**判定**：新建 PRD 还是修改现有？

**执行**：
1. 分析用户原始需求（通常模糊、不完整）
2. 输出 `01-requirement.md`：
   - 原始需求的理解和简单分析
   - 待澄清问题清单（向用户提问）
   - 功能边界初步定义
   - 新建/修改判定结果
   - 判定理由

**暂停点**：等待用户回答澄清问题后，才进入下一阶段。

---

### 阶段 2：方案调研

**目的**：基于项目上下文，搜索外部知识，形成可选方案。

**执行**：
1. 搜索行业最佳实践、竞品方案
2. 了解技术可行性（框架成本、实现难度）
3. 输出 `02-research.md`：

**方案对比维度**（视需求复杂度灵活调整）：

| 对比维度 | 说明 |
|---------|------|
| 实现复杂度 | 低/中/高，与当前技术栈匹配度 |
| 用户体验 | 完整度、流畅度 |
| 时间成本 | 预估工期 |
| 维护成本 | 后期迭代难度 |
| 扩展性 | 未来功能扩展空间 |

**方案示例参考**（非固定，视需求产出）：
- 方案 A（保守型）：最小实现，快速验证
- 方案 B（平衡型）：适中复杂度，较好体验
- 方案 C（完整型）：功能完整，实现成本较高

每个方案应包含：核心设计思路、优缺点分析、预估工作量、适用场景、与当前项目契合度。

**暂停点**：等待用户选择方案后，才进入下一阶段。

---

### 阶段 3：详细设计

**目的**：将选定方案细化为 PRD，清晰描述要做什么。

**原则**：
- 纯产品视角，聚焦"要做什么"
- 内容简洁扼要，突出重点
- 技术方面仅需标注：实现成本/难度

**执行**：
1. 编写 PRD 主体内容
2. 输出 `03-draft-prd.md`，内容视需求灵活调整，核心包含：
   - 元信息（版本、状态、作者、日期）
   - 功能描述（目标、核心功能点，简单清晰）
   - 交互设计（关键流程和界面）
   - 技术约束（引用 [[Architecture/xxx]]，标注实现成本/难度）
   - 验收标准
   - 依赖功能（双向链接 [[功能名_prd]]）

---

### 阶段 4：自我 Review

**目的**：检查方案合理性，发现问题。

**执行**：
1. 审查 `03-draft-prd.md`
2. 输出 `04-review.md`：
   - 矛盾点检查：功能间逻辑是否冲突
   - 遗漏点检查：边界情况、异常流程是否覆盖
   - 过度设计检查：是否超出当前阶段需要
   - 简化设计检查：是否过于简陋，影响用户体验
   - 技术可行性确认：架构约束是否满足
   - Review 结论：通过 / 需修改（列出修改点）

**暂停点**：如有修改点，返回阶段 3 修改后重新 review。

---

### 阶段 5：影响评估与关联修改

**目的**：确保新功能与项目整体融合。

**执行**：
1. 扫描现有功能文档（通过标题和简介筛选可能相关的）
2. 输出 `05-impact.md`：
   - 受影响功能清单
   - 每个受影响功能的修改建议
   - 需要同步更新的文档列表
   - 待办事项（转入 Doc/_todo.md）

**必须执行**：如有影响，创建或修改对应功能的 `_next.md` 或 `_prd.md`。

**暂停点**：如发现 breaking change 或需要用户决策的影响。

---

### 阶段 6：统一 Review

**目的**：整体一致性检查。

**执行**：
1. 汇总所有相关文档（新 PRD + 修改的 PRD）
2. 输出 `06-final-review.md`：
   - 功能间一致性检查
   - 数据流完整性检查
   - 用户体验连贯性检查
   - 最终结论：可发布 / 需调整

---

### 阶段 7：正式输出

**执行**：
1. 将 `03-draft-prd.md` 正式命名为 `{功能名}_prd.md`，放入 `Doc/`
2. 输出 `07-summary.md`：
   - 本次工作摘要
   - 所有生成/修改的文档清单
   - 遗留问题（如有）
   - 下一步建议

---

## 文档命名规范

过程中文档统一放在 `Doc/PRD-Process/{功能名}/`：

```
Doc/PRD-Process/
├── {功能名}/
│   ├── 00-context.md        # 项目上下文扫描
│   ├── 01-requirement.md    # 需求分析
│   ├── 02-research.md       # 方案调研
│   ├── 03-draft-prd.md      # PRD 草稿
│   ├── 04-review.md         # 自我 Review
│   ├── 05-impact.md         # 影响评估
│   ├── 06-final-review.md   # 统一 Review
│   └── 07-summary.md        # 工作总结
```

最终 PRD：`Doc/{功能名}_prd.md`

---

## 关键检查点

每个阶段完成后，必须自检：
- [ ] 是否建立了必要的 Obsidian 双向链接？
- [ ] 是否符合 document_rules.md 的文档分类？
- [ ] 是否评估了对其他功能的影响？
- [ ] 是否记录了待办事项？
